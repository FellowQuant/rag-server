## Accomplishments

- Worker pipeline (run_pipeline) implements the full parse->embed->upsert flow with SQLite and Qdrant rollback on failure, indexed_partial status for Docling PARTIAL_SUCCESS, and no ParsedChunk monkey-patching via parallel chunk_ids list

## Decisions Made

- chunk_ids generated as a parallel list to parsed_chunks in run_pipeline(); ParsedChunk dataclass is never monkey-patched with extra attributes like `_sqlite_id` -- this would bypass the dataclass contract
- Sync QdrantClient used inside worker process (not AsyncQdrantClient) -- worker has no asyncio event loop; the async client from QdrantStore (FastAPI-side) would deadlock without an event loop
- _get_sync_engine() strips "aiosqlite" from the FastAPI sqlite URL: `sqlite+aiosqlite:///` -> `sqlite:///` -- same DB file, different driver
- Document.indexed_at is set for both "indexed" and "indexed_partial" terminal statuses so callers can determine when indexing completed regardless of partial success
- WorkerManager.enqueue() uses put_nowait (non-blocking) -- callers must handle queue.Full and return 503
- Rollback deletes Chunk rows explicitly (not by deleting Document) so Document row survives with status="failed" and can be polled via GET /documents/{id}

- **Found during:** Task 2a (pipeline.py)
- **Issue:** The plan's _dispatch_parser called `converter.convert()` to check status and then called `parse_pdf(file_path, converter)` which calls `converter.convert()` internally again -- causing double-conversion. The plan comment said "to avoid double-converting" but still called parse_pdf after.
- **Fix:** Kept the two-call pattern but added a clear code comment explaining the tradeoff: parse_pdf owns all chunking logic and inlining it would duplicate significant code; double-convert is acceptable since Docling caches tokenization state. The pre-flight status check is minimal overhead versus inlining 80+ lines of parse_pdf logic.
