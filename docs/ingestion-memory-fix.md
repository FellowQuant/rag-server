## 1. Executive Summary

This report documents the full process of ingesting a **61-book quant finance
PDF library** (~18,210 pages) into a RAG server running on consumer hardware
(14 GB RAM, 8 GB VRAM). It covers the critical bug fix that made ingestion
viable, the bulk ingestion campaign, an anomalous 752-page book that required
special handling, and comprehensive post-ingestion validation.

**The double conversion bug.** The pipeline suffered from a bug that caused
every PDF to be fully converted by Docling's VLM pipeline **twice** per
document. The root cause was in `pipeline.py:_dispatch_parser()`, which called
`converter.convert()` to check the conversion status, then called `parse_pdf()`
which called `converter.convert()` again internally. Both `ConversionResult`
objects coexisted in memory simultaneously.

**The fix** eliminates the redundant first conversion by refactoring
`parse_pdf()` to return the conversion status (`is_partial`) alongside the
parsed chunks, allowing `_dispatch_parser()` to get both results from a single
conversion.

**The queue_42 anomaly.** One book (*Neural Networks for Algorithmic Trading
with MQL5*, 752 pages) consumed ~120 MB/page during Docling preprocessing —
24x the typical ~5 MB/page. Even after the double-conversion fix, the full book
OOM-killed the system at 94 pages in 46 seconds. The book was generated by
Help+Manual 8, which produces atypically dense embedded content. Solved by
splitting into 16 parts of 47 pages each.

**The Docling memory leak fix.** A separate investigation (documented in
`docs/docling-memory-leak.md`) identified that Docling's `keep_backend=True`
flag prevents per-page cleanup, and `_unload()` only closes native handles
without freeing Python-level data. The fix — `_teardown_conversion_result()`
in `pdf_parser.py` — explicitly breaks all references after chunk extraction.
Validated by ingesting 16 new finance papers (476 pages) sequentially with zero
restarts, reducing per-document memory retention from 200-500 MB to ~83 MB
(3-5x improvement).

**Final result:**
- 77 books ingested: 60 as single documents, 1 split into 16 parts, 16 new papers
- 92 documents, 107,079 chunks, 18,686 pages
- SQLite ↔ Qdrant: exact match (107,079 points, collection status: green)
- Search quality validated across 7 cross-book queries with reranker scores
  0.99–1.0

## 3. Problem Statement

During bulk ingestion of a 61-book quant finance PDF library (~15,000 total
pages), the RAG server worker process was consistently OOM-killed by the Linux
kernel. The failures exhibited three patterns:

1. **Large single documents:** Any PDF over ~250 pages would OOM-kill the worker
   during the VLM processing phase, regardless of batch position.
2. **Cumulative growth:** Even when processing smaller PDFs (<100 pages),
   sequential processing of 4-6 books would exhaust system RAM.
3. **No recovery:** After an OOM kill, documents would be left in `indexing`
   status permanently (no status rollback), and partial Qdrant data would remain
   without corresponding SQLite completion.

The server needed to ingest 61 PDFs ranging from 23 to 794 pages. With the
original code, only 8 books (all under 452 pages) had been successfully indexed
across multiple attempts.

## 4. Investigation

### 4.1 Initial Observations

Initial debugging focused on batch size and ingestion ordering, under the
hypothesis that the cumulative volume of uploaded files was causing OOM. Books
were re-ordered from smallest to largest and batch sizes reduced from 5 to 1.
The server continued to crash, including on an 88-page PDF -- ruling out the
"too many files" hypothesis.

Key observation: an 88-page PDF should not require anywhere near 14 GB of RAM.
The Docling VLM processes pages in batches of 2 (configured via
`docling_settings.perf.elements_batch_size = 2`) and should have bounded memory
usage. This pointed to a bug in our pipeline code rather than an inherent
Docling limitation.

### 4.2 Profiling Infrastructure

A lightweight RSS profiler was added to the worker process to measure memory at
each pipeline stage:

```python
def _log_mem(label: str) -> None:
    """Log current process RSS in MB for memory profiling."""
    rss_kb = 0
    try:
        with open(f"/proc/{os.getpid()}/status") as f:
            for line in f:
                if line.startswith("VmRSS:"):
                    rss_kb = int(line.split()[1])
                    break
    except Exception:
        pass
    logger.info("MEM [%s]: RSS=%.0f MB", label, rss_kb / 1024)
```

Probes were placed at: `pipeline-start`, `before-convert`, `after-convert`,
`after-convert-gc`, `after-parse`, `after-sqlite-write`, `after-qdrant-upsert`,
`pipeline-end`, `after-gc`.

### 4.3 Root Cause Analysis

Reading the `_dispatch_parser()` function revealed the core issue. The original
code flow for PDF processing was:

```
_dispatch_parser()
  |
  +--> converter.convert(file_path)    # FIRST full conversion
  |      produces ConversionResult #1 (status_result)
  |      ~500-3000 MB depending on page count
  |
  +--> parse_pdf(file_path, converter) # SECOND full conversion
         internally calls converter.convert(file_path)
         produces ConversionResult #2
         ~500-3000 MB depending on page count
```

Both `ConversionResult` objects existed in memory simultaneously: `status_result`
remained on the stack in `_dispatch_parser()` while `parse_pdf()` created its
own conversion result. Each `ConversionResult` contains:

- Per-page backend objects (PDF rasterization buffers)
- Per-page image tensors (for VLM inference)
- Layout predictions, table structure, formula/code recognition results
- The assembled `DoclingDocument` with all extracted content

For a 794-page PDF, each `ConversionResult` consumed approximately 3-4 GB of
RAM. Two copies simultaneously: **6-8 GB consumed by conversion data alone**,
leaving no headroom for the embedder, SQLite, Qdrant client, or OS.

The original code contained a misleading comment:

```python
# Double-convert is acceptable since converter caches tokenisation state.
```

This comment was incorrect. `DocumentConverter.convert()` does **not** cache
conversion results between calls. Each call performs the full VLM pipeline from
scratch: page rasterization, layout detection, table structure recognition,
code/formula enrichment. The "tokenisation state" mentioned in the comment does
not exist in Docling's architecture -- the converter is stateless between
`convert()` calls.

The reason for the double conversion was architectural: `_dispatch_parser()`
needed the `ConversionStatus` (to detect `PARTIAL_SUCCESS`), but `parse_pdf()`
returned only chunks, not the status. Rather than modifying the interface, the
original code performed a redundant first conversion just to check the status.

## 5. The Fix

### 5.1 Changes to `pdf_parser.py`

**File:** `src/rag_server/ingestion/parsers/pdf_parser.py`

The `parse_pdf()` function signature was changed to return the conversion status
alongside the chunks:

**Before:**
```python
def parse_pdf(file_path, converter) -> list[ParsedChunk]:
    ...
    return raw_chunks
```

**After:**
```python
def parse_pdf(file_path, converter) -> tuple[list[ParsedChunk], bool]:
    ...
    return raw_chunks, is_partial
```

The function now internally checks `ConversionStatus.PARTIAL_SUCCESS` and
returns an `is_partial` boolean. Return values:

| Conversion Status | Return Value |
|-------------------|--------------|
| `SUCCESS`         | `(chunks, False)` |
| `PARTIAL_SUCCESS` | `(chunks, True)` |
| `FAILURE`         | `([], False)` |

This is a minimal, safe change: `parse_pdf()` already had access to
`result.status` internally. The only modification was exposing it in the return
type instead of discarding it.

### 5.2 Changes to `pipeline.py`

**File:** `src/rag_server/worker/pipeline.py`

The `_dispatch_parser()` PDF branch was simplified from a two-step
convert-then-parse to a single `parse_pdf()` call:

**Before (double conversion):**
```python
if file_format == "pdf":
    from docling.datamodel.base_models import ConversionStatus
    from rag_server.ingestion.parsers.pdf_parser import parse_pdf

    if converter is None:
        from rag_server.ingestion.parsers.pdf_parser import make_converter
        converter = make_converter(use_gpu=True)

    # FIRST CONVERSION -- just to check status
    status_result = converter.convert(str(file_path), raises_on_error=False)
    is_partial = (status_result.status == ConversionStatus.PARTIAL_SUCCESS)

    if status_result.status == ConversionStatus.FAILURE:
        logger.error("Docling: total failure on %s -- %s", file_path, status_result.errors)
        return [], False

    if is_partial:
        logger.warning("Docling: partial success on %s -- %s", file_path, status_result.errors)

    # SECOND CONVERSION -- inside parse_pdf()
    chunks = parse_pdf(file_path, converter)
    return chunks, is_partial
```

**After (single conversion):**
```python
if file_format == "pdf":
    from rag_server.ingestion.parsers.pdf_parser import parse_pdf

    if converter is None:
        from rag_server.ingestion.parsers.pdf_parser import make_converter
        converter = make_converter(use_gpu=True)

    _log_mem("before-convert")
    chunks, is_partial = parse_pdf(file_path, converter)
    _log_mem("after-convert")

    # Reclaim Docling per-document memory (page backends, image caches).
    gc.collect()
    try:
        import torch
        torch.cuda.empty_cache()
    except Exception:
        pass
    _log_mem("after-convert-gc")

    return chunks, is_partial
```

Additional changes to `run_pipeline()`:
- Added `gc.collect()` at the end of each pipeline run to reclaim Docling
  per-document objects
- Added `torch.cuda.empty_cache()` to release CUDA memory after conversion
- Added `_log_mem()` probes at all pipeline stages for ongoing profiling

### 5.3 Correctness Guarantees

The fix preserves all existing behavior:

1. **Status detection:** `PARTIAL_SUCCESS` detection moved from
   `_dispatch_parser()` into `parse_pdf()`, which already had access to the
   status. The `indexed_partial` status flow is unchanged.
2. **Failure handling:** `FAILURE` detection remains in `parse_pdf()` (it
   already handled this case by returning an empty list). Now it returns
   `([], False)` instead of `[]`.
3. **Chunk output:** The chunks produced are identical -- same parser, same
   Docling configuration, same chunking logic. Only the number of conversions
   changed (2 -> 1).
4. **Single caller:** `parse_pdf()` is called from exactly one location
   (`_dispatch_parser()`), so the signature change has no ripple effects.
5. **Non-PDF formats:** LaTeX and Jupyter parsers are unaffected (they don't
   use Docling or `converter.convert()`).

## 7. Bulk Ingestion Campaign

### 7.1 Strategy and Tooling

After validating the double-conversion fix on 4 books (Section 6.2), the
remaining 43 books were ingested. The strategy accounted for Docling's
per-document memory leak (Section 10.1):

1. **One book at a time** with full server restart between each document,
   eliminating memory leak accumulation.
2. **Automated shell scripts** (`ingest_small.sh`, `ingest_remaining.sh`,
   `ingest_q42.sh`) that: start the server, wait for health check, upload
   the PDF via the `/api/v1/documents` endpoint, poll SQLite for completion,
   stop the server, repeat.
3. **External RAM monitor** (`scripts/ram_monitor.sh`): a lightweight
   independent process that polls `free -m` and the heaviest `rag-server`
   process RSS every 5 seconds, writing to a persistent log file. Survives
   OOM kills since it runs as a separate process.

### 7.2 Main Batch: 43 Books

The 43 remaining books were processed using `ingest_small.sh`. All books
completed successfully except one — `queue_42.pdf` (752 pages), which
OOM-killed the system on every attempt.

Two additional books required separate processing due to file naming issues
(Unicode characters in filenames break `curl` uploads):
- `queue_24.pdf` — 284 pages, completed with status `indexed_partial` (only 42
  of 284 pages parsed; see Section 10.5)
- `queue_43.pdf` — 771 pages, completed successfully

### 7.3 The queue_42 Problem: 752-Page Anomaly

**Book:** *Neural Networks for Algorithmic Trading with MQL5* (752 pages)
**PDF Generator:** Help+Manual 8 (identified from PDF metadata)

This book exhibited anomalous memory consumption that far exceeded any other
book in the library, including books with more pages. The failure timeline:

| Attempt | Configuration | Pages Processed | Time to OOM | Peak RSS |
|---------|--------------|-----------------|-------------|----------|
| 1       | Full 752p, batch=2 | ~94 pages | ~46 seconds | >11 GB (system froze) |
| 2       | Full 752p, batch=1 | ~94 pages | ~46 seconds | >11 GB (system froze) |
| 3       | Half 376p | ~94 pages | ~46 seconds | >11 GB (system froze) |

**Key finding:** Reducing `elements_batch_size` from 2 to 1 had **no effect**
because the crash occurred during Docling's page preprocessing/layout detection
phase, which runs **before** the CodeFormulaV2 VLM inference that
`elements_batch_size` controls. The crash was caused by per-page memory
consumption during the preprocessing stage itself.

**Memory per page comparison:**

| Book Type | Typical MB/page | queue_42 MB/page |
|-----------|----------------|-----------------|
| Text-heavy finance PDFs | ~5 MB | — |
| Formula-dense PDFs | ~40-100 MB | — |
| queue_42 (Help+Manual 8) | — | **~120 MB** |

The Help+Manual 8 PDF generator produces atypically dense embedded content per
page, causing Docling's page preprocessing to allocate ~24x more memory than
typical born-digital finance PDFs.

## 9. Other Bugs Discovered

### 9.1 `pkill` Pattern Mismatch (Critical)

**Impact:** All three ingestion scripts (`batch_ingest.sh`, `ingest_small.sh`,
`ingest_heavy.sh`) failed to kill the server process between batches.

**Root cause:** The scripts used `pkill -f "rag_server"` (underscore), but the
actual process name is `rag-server start` (hyphen). The regex never matched, so
old server processes survived every "restart".

**Consequences:**
- New server instances could not bind to port 8001 (already in use by the old
  process), and silently failed.
- Code changes were never loaded -- all profiling data came from a single stale
  process running the old code.
- Memory profiling results were misleading until this was discovered.

**Fix:** Changed all scripts to `pkill -f "rag.server"` (regex dot matches both
hyphen and underscore).

**Files changed:** `scripts/batch_ingest.sh`, `scripts/ingest_small.sh`,
`scripts/ingest_heavy.sh`

### 9.2 Wrong Health Endpoint in Scripts

**Impact:** `batch_ingest.sh` polled `/api/v1/health` to detect server
readiness, but the actual endpoint is `/health`. The server appeared to never
start (all health checks returned 404), causing the script to timeout after
270 seconds on every batch.

**Fix:** Changed to `/health` in all scripts.

### 9.3 Wrong Server Start Command in Scripts

**Impact:** `batch_ingest.sh` used `python -m rag_server` to start the server,
but this fails with `No module named rag_server.__main__`. The correct command
is `uv run rag-server start` (CLI entry point defined in `pyproject.toml`).

**Fix:** Changed to `uv run rag-server start` in all scripts.

### 9.4 RAM Monitor Tracking Wrong Process

**Impact:** The initial RAM monitor used `pgrep -f "multiprocessing"` to find
the worker process. This matched Python's `multiprocessing.resource_tracker`
(4 MB RSS), not the actual ingestion worker (8+ GB RSS). All early RAM
measurements were off by three orders of magnitude.

**Root cause:** The worker subprocess is spawned by the FastAPI server and
inherits the `rag-server` process name, not a `multiprocessing`-specific name.

**Fix:** Changed to find the heaviest process matching `pgrep -f "rag.server"`
by iterating all matching PIDs and selecting the one with the highest
`VmRSS` from `/proc/<pid>/status`.

### 9.5 Orphaned Data in SQLite and Qdrant

**Impact:** Documents that were deleted and re-ingested (during debugging and
retry cycles) left behind orphaned data:
- **SQLite:** 2,062 chunks from 1 deleted document ID
- **Qdrant:** 14,753 points from 22 deleted document IDs

**Root cause:** The delete-and-reingest workflow cleaned the `documents` table
but did not cascade to the `chunks` table or Qdrant collection. There is no
foreign key cascade or Qdrant cleanup trigger.

**Fix:** Manual cleanup during validation (Section 8.2, 8.3). A future
improvement would add cascade deletes and a Qdrant cleanup step to the document
deletion endpoint.

## 10. Known Unresolved Issues

### 10.1 Docling Per-Document Memory Leak — RESOLVED

Docling's `DocumentConverter.convert()` does not fully release per-document data
structures after conversion. The `keep_backend=True` flag (required for formula
and code enrichment) prevents per-page cleanup during `_integrate_results()`,
and `_unload()` only closes native handles without freeing Python-level data
(image caches, parsed pages, DoclingDocument, predictions).

**Fix (2026-04-12):** `_teardown_conversion_result()` in `pdf_parser.py`
explicitly breaks all references inside ConversionResult after chunk extraction.
See `docs/docling-memory-leak.md` for full investigation and test results.

**Validation:** 16 PDFs (476 pages) ingested sequentially with zero restarts.
Per-document retention reduced from 200-500 MB to ~83 MB average (3-5x
improvement). RSS went down on one document, proving cross-document reclamation.
Projected safe capacity: ~30-40 docs per session before restart recommended.

### 10.2 No Memory Cap on Ingestion Worker

The worker subprocess has no memory limit (`ulimit`, cgroup, or application-
level). If Docling allocates beyond system RAM, the Linux OOM killer terminates
the process without graceful shutdown, leaving documents in `indexing` status
with no rollback.

**Potential mitigations (not implemented):**
- Set `ulimit -v` or cgroup memory limits on the worker process
- Implement a memory watchdog thread that aborts processing if RSS exceeds a
  configurable threshold
- Use `resource.setrlimit(resource.RLIMIT_AS, ...)` in the worker

### 10.3 Temporary File Loss on Reboot

Uploaded PDFs are stored in `/tmp/ingest-upload/`, which is cleared on reboot.
During this investigation, OOM-kills triggered system instability and reboots,
causing uploaded files to be lost multiple times. Files had to be re-copied from
the source directory.

**Recommendation:** Use a persistent directory (e.g., `data/uploads/`) for
staging files intended for bulk ingestion.

### 10.4 Empty Content Chunks (1.5%)

1,548 chunks have NULL or empty `content` fields. Breakdown: 1,056 code, 491
table, 1 formula. These are Docling extraction failures where the VLM detected a
region but failed to extract text. The chunks have correct metadata and their
empty embeddings do not materially affect search quality (low similarity to any
real query).

This is an upstream Docling limitation. No fix available from our pipeline code.

### 10.5 Partial Document: queue_24.pdf

*Building Winning Algorithmic Trading Systems* (284 pages) was ingested with
status `indexed_partial`. Docling reported `PARTIAL_SUCCESS` and only extracted
42 of 284 pages. The remaining 242 pages may contain content that Docling's
layout model could not process (complex charts, scanned appendices, or
non-standard page layouts).

**Potential mitigation (not attempted):** Re-ingest with OCR enabled
(`do_ocr = True`), at the cost of ~1 GB additional VRAM for the RapidOCR model.
Alternatively, split the book and re-ingest parts to isolate problematic pages.

## 11. Operational Recommendations

For bulk ingestion on this hardware (14 GB RAM, 8 GB VRAM):

1. **Sequential ingestion without restarts (up to ~30 docs):** With the
   `_teardown_conversion_result()` fix, sequential processing is reliable
   for ~30-40 documents before restart is recommended. For very large batches,
   restart every ~30 documents or when worker RSS exceeds ~6 GB.

2. **Split books over 500 pages:** Books generated by non-standard PDF
   producers (Help+Manual, etc.) may consume 100+ MB/page. Split into parts
   of 50 pages or fewer to stay within safe memory bounds.

3. **Run the external RAM monitor:** `scripts/ram_monitor.sh` writes to a
   persistent log that survives OOM kills. Start it before any bulk ingestion.

4. **Use persistent staging directories:** Copy PDFs to a non-`/tmp` location
   before ingestion to survive reboots.

5. **Budget per-document peak RSS:**
   - Text-heavy PDFs: ~5-10 MB/page peak
   - Formula-dense PDFs: ~40-100 MB/page peak
   - Anomalous generators (Help+Manual): ~120 MB/page peak
   - Add ~1,200 MB baseline (embedder + models + framework)

6. **Post-ingestion validation:** After bulk ingestion, verify SQLite ↔ Qdrant
   consistency. The delete/re-ingest cycle does not cascade cleanly and will
   produce orphaned data that must be manually cleaned.
